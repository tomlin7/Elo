## 1. Objective & Strategy

Our initial release proved we could deliver a sub-40ms synchronized math duel on Android. But a fast engine alone does not create a habit. To transition Elo from a technical showcase into a sticky competitive platform, we have identified four key pillars for this update:

1. **Retention** — Keep players coming back with Daily Streaks and XP leveling.
2. **Social Growth** — Let players invite friends with custom private rooms.
3. **Monetization Foundation** — Introduce a cosmetic economy via an in-game credit system.
4. **Trust & Security** — Enforce server-side answer validation and input cadence profiling.

## 2. Progression Framework

### Daily Streak Tracker

The streak system operates on a rolling 24-hour check-in window. Completing at least **3 ranked matches** in a calendar day maintains the streak. As milestones accumulate, the home screen's streak flame icon visually escalates in intensity:

| Milestone | Flame Color |
| --- | --- |
| Day 1–6 | Bronze |
| Day 7–29 | Silver |
| Day 30–99 | Gold |
| Day 100+ | Cosmic Purple |

### XP & Account Leveling

Every completed match awards performance-based experience points using the following formula:

$$\text{XP}_{\text{Earned}} = (\text{MatchOutcome} \times 100) + (\text{CorrectAnswers} \times 5) + (\text{MaxStreak} \times 2)$$

Advancing account levels unlocks profile badge borders and custom titles displayed in matchmaking lobbies and on the global leaderboard.

## 3. Custom Rooms & Private Lobbies

### Direct Friendly Duels

Players can generate a **6-digit numeric room code** with a single tap on "Create Private Room." The code can be shared via the native Android share sheet (`Share.share`) as a deep link to instantly drop a friend into the waiting lobby.

Private rooms intentionally **bypass global Elo rating calculations** but maintain a separate head-to-head record for bragging rights.

### Modular Rulesets

Room owners have full control over the match ruleset before kicking off the game:

* **Operations**: Restrict questions to only multiplication/division, pure addition/subtraction, or enable absolute value problems.
* **Round Duration**: Choose from 30s, 60s, or 120s sudden-death bounds.

## 5. Technical Architecture

### 5.1 UI Theme Engine

The cosmetic system requires zero-overhead theme switching. Theme values are injected via an atomic **Zustand store** (`useThemeStore`) that passes hex configurations directly into **React Native Reanimated** shared styling hooks:

```ts
const animatedStyle = useAnimatedStyle(() => ({
  backgroundColor: themeStore.primaryColor.value,
  borderColor: themeStore.accentColor.value,
}));
```

This approach avoids triggering a full React reconciliation cycle, keeping the JS thread free and ensuring UI settle times stay **under 5ms** during theme swaps.

### 5.2 Offline Sync via MMKV

To ensure the app feels snappy even with poor connectivity, all persistent player state is cached locally using **MMKV**—a native C++ key-value storage engine that is synchronous and up to 30× faster than `AsyncStorage`:

| State Type | Storage Strategy |
| --- | --- |
| Player profiles & titles | MMKV (synchronous read/write) |
| Active streak counters | MMKV |
| Unlocked cosmetics catalog | MMKV |
| Pending match backlog | MMKV |

## 6. Server-Side Anti-Cheat

### Input Cadence Profiling

The Rust game server now records the elapsed time in milliseconds between every numeric keyboard input using `ClientAction.timestamp`. A statistical model flags sessions where answers to multi-digit problems are consistently submitted faster than the human reaction floor:

```
Flag threshold: < 120ms per multi-digit evaluation (sustained over 5+ consecutive inputs)
```

Flagged matches are quarantined for manual review or automatic rating rollback.

### Client-Server Answer Scrutiny

Clients cannot self-report correctness. The server owns the master question generation seed and evaluates every submitted answer against the authoritative result. There is no `isCorrect` boolean in the client-to-server contract—only the raw answer string.

## 7. Extended Protobuf Schema

The network data contract for this update introduces three new message types:

```protobuf
syntax = "proto3";

package elo.v2;

enum RoomType {
  ROOM_TYPE_RANKED = 0;
  ROOM_TYPE_PRIVATE = 1;
}

message PlayerIdentity {
  string player_id = 1;
  string username = 2;
  int32 elo_rating = 3;
  string selected_theme_id = 4;
  string active_title = 5;
}

message CreateCustomRoomRequest {
  string host_player_id = 1;
  bool allow_division = 2;
  int32 custom_duration_seconds = 3;
}

message MatchSecurityLog {
  string action_sequence_id = 1;
  int64 keystroke_delta_ms = 2; // Time between consecutive keystrokes
  string input_value = 3;
}
```
